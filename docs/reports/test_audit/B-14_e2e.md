## Recit du framework via ce packet

### Episode 1 : Interface Web Playwright E2E (2026-01 a 2026-03)

Les fichiers `demos/` et `python/test_*interface*.py` cristallisent les **tests E2E de l'interface web** via Playwright :
- `demos/test_playwright_setup.py` + `_fixed.py` : validation que Playwright est disponible
- `demos/test_webapp_interface_demo.py` (6 tests) : interaction avec la SPA React (load, examples, clear, analyze, validation, responsive)
- `python/test_simple_demo.py` (3 tests) : app load, API connectivity, navigation tabs
- `python/test_argument_reconstructor.py` (5 tests) : workflow de reconstruction d'arguments via UI
- `python/test_fallacy_detector.py` (4 tests) : detection de sophismes via UI
- `python/test_validation_form.py` (4 tests) : formulaire de validation logique via UI
- `python/test_webapp_homepage.py` (1 test) : titre et header de la page d'accueil
- `web_api/test_interface_simple_playwright.py` (7 tests) : tests Playwright de l'interface simple

**Arc temporel** : ces tests sont contemporains de la **Phase 3 "authentique"** du projet. Ils testent l'interface web React (9 tabs) via Playwright avec demarrage automatique des serveurs (Starlette + FastAPI). La plupart des tests interagissent avec l'UI et verifient les reponses API. **Etat actuel : MIXED** — les tests UI testent le frontend React qui est servie par Starlette (devenu proxy via A-14 Option 1). Les endpoints API testes (analyze, fallacies, validate) sont cables dans FastAPI.

### Episode 2 : Framework Builder et Dung (2026-02)

`python/test_framework_builder.py` (5 tests), `python/test_api_dung_integration.py` (1 test) — cristallisent les tests E2E du **framework builder** (creation de cadres d'argumentation Dung via UI) et l'**API Dung** (analyse de frameworks via endpoint REST).

**Arc temporel** : ces tests documentent la transition entre l'interface standalone et l'API REST. `test_framework_builder.py` teste la creation de frameworks via l'UI React (onglet Framework), tandis que `test_api_dung_integration.py` teste directement l'API backend. **Etat actuel : WIRED** — `dung_extensions` est registree dans CapabilityRegistry avec invoke callable `_invoke_dung_extensions` et state writer `_write_dung_extensions_to_state`. L'endpoint API existe dans le router FastAPI.

### Episode 3 : Integration Workflows (2026-03)

`python/test_integration_workflows.py` (5 tests, 2 skip) — cristallise les **workflows d'integration complets** :
- Full argument analysis workflow (analyze → fallacies → validate)
- Framework-based validation workflow
- Cross-tab data persistence
- Performance stress workflow
- Integration suite health check

**Arc temporel** : ce fichier (687 LOC, le plus grand des tests E2E) documente l'ambition d'une **suite d'integration E2E complete** traversant toutes les tabs de l'interface. Les 2 tests skippees marquent les limites actuelles (performance stress et health check requierent serveur actif). **Etat actuel : WIRED** — les workflows testes correspondent aux capabilities pipeline (fallacy_detection, dung_extensions, validation logique).

### Episode 4 : Web API Management (2026-02 a 2026-05)

`web_api/test_interfaces_integration.py` (13 tests), `web_api/test_management_scripts.py` (11 tests) — cristallisent l'**infrastructure de gestion** :
- Coexistence des interfaces (directories, ports, dependencies, shared resources)
- Management scripts (health check, start/stop, port detection, ServiceManager)
- Integration environment (dependencies, logging, configuration)

**Arc temporel** : ces tests sont des **tests d'infrastructure** qui ne testent pas le pipeline metier mais l'environnement d'execution (directories, scripts, configuration). Ils sont essentiels pour la validation du deploiement. **Etat actuel : DEROGATION** — infrastructure critique pour le deploiement, pas du wiring pipeline.

## Narratif global

### Sante globale — 61% justifie, 26% perime

Le ratio 61% justifie est le **plus bas** de tous les audits B (B-01 a B-13 etaient entre 85-89%). Cela s'explique par la nature des tests E2E :
- Ils dependent fortement de l'**interface React** qui a evolue (9 tabs, 3-tier hybrid, hierarchical detection)
- La transition A-14 (Starlette→proxy) casse les tests qui supposent un backend Starlette
- Les 3 scripts non-pytest (1299 LOC) sont du code mort pour pytest (non collectes)
- Les tests d'infrastructure (web_api/) sont la partie la plus saine (24 tests, 100% justifies)

### Points d'attention

1. **3 scripts non-pytest** (`test_interface_web_complete.py`, `test_phase3_web_api_authentic.py`, `test_react_interface_complete.py`) : 1299 LOC de logique de test qui n'est **jamais executee** par pytest. Ces fichiers pourraient etre convertis en tests pytest ou archives.

2. **`test_playwright_setup.py`** est un doublon exact de `test_playwright_setup_fixed.py`. L'original devrait etre supprime.

3. **Tests UI perimes** : `test_argument_reconstructor.py`, `test_validation_form.py`, `test_fallacy_detector.py` testent des composants UI qui ont evolue. Ils sont fragiles et generent des faux positifs/negatifs.

4. **`test_webapp_api_investigation.py`** (4/6 tests skippees) : teste les endpoints mobile avec les bugs connus (audit A-15). La validite des tests restants est limitee.
